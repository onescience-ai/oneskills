## 一、数据集核心基础信息（代码生成前置知识）

### 1.1 数据集唯一标识
- 全称：ShapeNetCore 汽车子集（以下简称 ShapeNet-car）
- 唯一主键：`model_id`
- 建议保留标识：`synsetId`、`model_id`、原始目录名
- 数据类型：三维 CAD 形状网格、材质文件、纹理图片、渲染图、类别元信息
- 官方数据源：ShapeNet / ShapeNetCore 数据分发体系及其公开镜像
- 数据规模：ShapeNetCore 覆盖 55 个常见类别、约 51,300 个模型；本文档聚焦汽车类别

### 1.2 任务对象与解析优先级

| 任务对象 | 代码标识 | 解析优先级 | 处理逻辑 |
|---|---|---|---|
| 网格几何 | MESH | 高 | 默认读取 OBJ 顶点、面片、法向和几何包围盒，是最核心的数据主体 |
| 材质纹理 | MATERIAL | 中 | 关联 MTL、PNG/JPG 纹理文件，用于材质分析、渲染和跨模态训练 |
| 多视图渲染 | RENDER | 中 | 按视角目录读取渲染图，用于检索、重建和图像到形状任务 |
| 类别元信息 | META | 高 | 读取 taxonomy、synset 与类别映射，确保标签和目录可追溯 |

## 二、数据格式规范与字段映射（必解析字段清单）

### 2.1 主格式与核心字段

| 格式类型 | 字段含义 | 代码变量名 | 数据类型 | 是否必填 | 缺失处理 |
|---|---|---|---|---|---|
| OBJ | 顶点、面片、法向 | `mesh_obj` | 文本/网格 | 是 | 缺失则主样本无效 |
| MTL | 材质与纹理引用 | `mesh_mtl` | 文本 | 否 | 缺失时保留纯几何 |
| PNG/JPG | 纹理或渲染图像 | `image_path` | 路径 | 否 | 缺失则标记无纹理 |
| taxonomy | 类别映射与 synset 关系 | `class_meta` | JSON/CSV | 是 | 缺失则标记未知类别 |

### 2.2 兼容格式或目录级组织

| 行标识/目录项 | 字段含义 | 代码变量名 |
|---|---|---|
| `model_normalized.obj` | 标准化汽车网格 | `mesh_path` |
| `model_normalized.mtl` | 材质文件 | `mtl_path` |
| `rendering/*.png` | 多视图渲染图 | `render_dir` |

### 2.3 数据组织说明
- 主数据应围绕 OBJ/MTL 建立解析器，不建议在首阶段直接只保留点云，否则会丢失面片拓扑与材质引用关系。
- 如果下游模型需要点云、体素或多分辨率网格，应将这些派生表示作为新增结果输出，而不是覆盖原始主格式。
- 渲染图通常不是样本识别的唯一真源，代码实现时应允许“有网格、无渲染图”的情况正常通过。

## 五、解析逻辑与遍历规则（分层执行步骤）

### 5.1 单条目解析步骤（智能体按顺序生成代码）
- 根据 `model_id` 定位样本目录并确认 OBJ 文件存在。
- 解析顶点、面片、法向和网格的包围盒、中心点、尺度等统计量。
- 如存在 MTL，则解析材质名、纹理路径和多材质分段关系。
- 如存在 `renderings` 目录，则按视角顺序读取图像路径与尺寸信息。
- 将 `geometry`、`material`、`render`、`meta` 四类字段统一写入结果字典。

### 5.2 多样本或多分量处理规则
- 同一模型的几何、材质和图像资源必须通过 `model_id` 对齐，禁止跨目录拼接资源。
- 批量导出时应保持原始 `model_id` 不变，同时可另外生成整型索引映射表，便于训练时快速随机访问。

## 六、校验规则与异常处理（判断条件+分支逻辑）

### 6.1 必做校验（代码必须实现，顺序执行）
- OBJ 文件必须存在且顶点数、面片数均大于 0。
- 材质文件中引用的纹理路径应可在相对路径下解析到真实文件。
- taxonomy 与 synset 映射应能给出稳定的汽车类别标识。

### 6.2 异常处理分支（固定捕获+处理逻辑）

| 异常类型 | 触发场景 | 代码处理逻辑 |
|---|---|---|
| `FileNotFoundError` | OBJ 或目录缺失 | 跳过当前 `model_id` 并记录 missing 样本 |
| `ValueError` | 空网格、坏拓扑、顶点异常 | 标记坏样本，不中断整体任务 |
| `OSError` | 图像或纹理解码失败 | 忽略图像资源但保留可用几何 |

## 七、数据过滤与清洗规则（剔除条件清单）
代码必须在解析过程中实时过滤，禁止等全部数据读取完成后再统一做清洗。这样做可以减少内存占用，也能更快发现输入源中的结构性问题。

- 过滤空网格、零面积面片和明显损坏的拓扑样本。
- 可按顶点数阈值筛掉过粗或极端复杂模型，保持训练集分布稳定。
- 如果任务与材质无关，可在解析后只保留纹理存在标志而不加载整张图。

## 八、批量调度与存储规范（路径+命名+格式）

### 8.1 目录规范（代码自动创建，禁止修改路径）
- 建议目录结构：`./storage/shapenet_car/{model_id}.json` 或 `{model_id}.npz`
- 原始 OBJ/MTL 不建议复制到结果目录，避免重复占用空间。
- 若同时导出点云、体素或渲染摘要，应放到 `geometry`、`voxel`、`render` 子目录。

### 8.2 结果格式规范（固定JSON结构）

```json
{
  "dataset_name": "ShapeNet-car",
  "sample_id": "model_id",
  "meta": {"category": "car", "status": "ok"},
  "payload": {
    "obj_path": "./models/model_normalized.obj",
    "num_vertices": 12345,
    "num_faces": 24680,
    "has_texture": true
  }
}
```

### 8.3 批量调度规则
- 批量处理模式为 FIFO，按 `model_id` 列表顺序执行。
- 已生成有效 JSON/NPZ 时可直接跳过，支持断点续跑。
- 图像资源建议在 QA 阶段抽样读取，主流程只记录路径和尺寸摘要。
