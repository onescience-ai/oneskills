# Onescience 任务测试技能

执行 Onescience 相关任务测试时，先判断任务类型，再读取对应 reference，最后按统一闭环完成测试、日志收集和结果汇报。

## 任务分流

先识别当前任务属于哪一类：

- 模型实现测试：检查模型结构、实例化、前向传播、梯度、设备兼容性、集群提交与日志分析。
- 气象数据 Pipeline 测试：检查 DataPipe / Dataset 的格式适配、时间窗口、区域裁剪、PyTorch 集成和性能约束。
- 生物信息学工具测试：检查生物工具的输入输出、领域逻辑、与 Onescience 框架集成及可执行性。

如果任务跨多个类别，先测主任务，再补测其依赖模块；不要把不同类型的检查点混成一个模糊流程。

## 参考文件导航

按任务类型读取最少必要的 reference：

- 模型测试：读取 `references/model_test.md`
- 气象 DataPipe 测试：读取 `references/earth_datapipe.md`
- 生物信息学工具测试：读取 `references/bio_tool.md`

只有在当前任务确实需要时才继续深读 reference；不要一次性加载全部参考文档。

## 模型测试工作方式

处理模型测试任务时：

1. 先读取模型文件，提取类名、构造参数、`forward` 签名、输入输出 shape、依赖文件和运行资源。
2. 检查是否已有 `test_model_<name>.py`、`test_<name>.py`、`test_<name>.sh` 或 `tests/` 中的现成测试。
3. 若无测试，则按 `references/model_test.md` 中的固定模板生成测试文件；不要随意改写模板结构。
4. 生成或校对任务执行脚本，保证测试文件名、执行命令和提交文件清单一致。
5. 需要集群执行时，通过 scnet MCP 提交文件和任务，收集 `.out` / `.err` 日志。
6. 输出结构化测试报告；若失败，先定位失败点和根因，再给出最小修复建议。

## DataPipe / 工具测试工作方式

处理非模型任务时：

1. 先确定领域和输入数据类型。
2. 针对气象 DataPipe，重点核对数据格式读取、变量顺序、时间窗口、训练验证划分、区域切片、Dataset 接口、懒加载和错误处理。
3. 针对生物信息学工具，重点核对 Onescience 模块集成、输入格式、核心算法调用、参数化、日志与输出规范。
4. 若用户要求“测试”，优先做可执行验证；若缺少真实数据或环境，再退回静态审查并明确说明限制。

## 标准执行闭环

无论哪类任务，都遵循下面的顺序：

1. 识别任务类型。
2. 读取对应 reference。
3. 提取被测对象的关键接口、依赖和运行条件。
4. 查找已有测试资产并尽量复用。
5. 补齐测试脚本、执行脚本或验证步骤。
6. 本地检查文件一致性。
7. 需要时提交到 scnet 集群执行。
8. 收集日志和结果。
9. 输出测试报告、风险点和后续建议。

## 输出要求

测试完成后至少输出以下内容：

- 测试对象和任务类型
- 使用的 reference 文件
- 实际执行的测试项
- 结果汇总，包含 PASS / FAIL 或可执行 / 不可执行结论
- 关键错误信息或风险说明
- 若已提交 scnet，补充任务提交状态和日志结论

如果测试未能真正执行，也必须明确原因，例如：

- 缺少依赖
- 缺少输入数据
- scnet 任务提交失败
- 运行环境不满足

## 约束

- 优先复用现有测试，不重复生成等价文件。
- 不要自动安装依赖；缺什么就报告什么。
- 不要在没有证据的情况下声称“测试通过”。
- 模型测试中若 reference 明确要求保留模板，则只填充占位符，不改模板主体。
- 保留完整错误堆栈和关键信息，不截断核心报错。
- 若任务未成功提交到 scnet，则停止结果推断，先报告提交失败。
