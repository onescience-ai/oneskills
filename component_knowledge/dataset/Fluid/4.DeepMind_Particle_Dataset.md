## 一、数据集核心基础信息（代码生成前置知识）

### 1.1 数据集唯一标识
- 全称：DeepMind Learning to Simulate 粒子动力学数据集
- 唯一主键：`trajectory_id`
- 数据类型：粒子位置、粒子类型和上下文组成的时序物理仿真数据
- 官方数据源：`google-deepmind/deepmind-research` 中的 `learning_to_simulate` 项目
- 数据规模：官方子集包括 `WaterDrop`、`Water`、`Sand`、`Goop`、`MultiMaterial`、`Water-3D` 等，通常由 `metadata.json` 与 `train/valid/test.tfrecord` 组成

### 1.2 任务对象与解析优先级

| 任务对象 | 代码标识 | 解析优先级 | 处理逻辑 |
|---|---|---|---|
| 粒子轨迹 | TRAJ | 高 | 读取粒子位置序列、粒子类型和上下文，是最核心监督对象 |
| 元信息统计 | META | 高 | 读取边界范围、默认邻接半径和标准化统计 |
| 调试样本 | DEBUG | 中 | 如 `WaterDropSample`，用于快速验证管线 |

## 二、数据格式规范与字段映射（必解析字段清单）

### 2.1 主格式与核心字段

| 格式类型 | 字段含义 | 代码变量名 | 数据类型 | 是否必填 | 缺失处理 |
|---|---|---|---|---|---|
| `metadata.json` | 序列长度、边界和统计量 | `meta` | JSON | 是 | 缺失则停止 |
| TFRecord | 轨迹样本 | `record` | 二进制 | 是 | 损坏则重下 |
| `subset name` | 子数据集标识 | `subset` | 字符串 | 是 | 缺失则无法区分任务 |

### 2.2 兼容格式或目录级组织

| 行标识/目录项 | 字段含义 | 代码变量名 |
|---|---|---|
| `position` | 粒子位置序列 | `positions` |
| `particle_type` | 粒子类型 | `particle_type` |
| `context` | 全局上下文 | `global_context` |

### 2.3 数据组织说明
- 主格式由 `metadata` 和 `TFRecord` 共同构成，`metadata` 记录边界范围、默认邻接半径、标准化统计等关键参数。
- 同一代码框架下的不同子集在材料类型、维度和运动模式上差异很大，因此读取器必须保留 `subset_name`。
- 工程上通常需要把完整轨迹切成定长窗口，供 `one-step` 学习器或 `rollout` 评测器使用。

## 五、解析逻辑与遍历规则（分层执行步骤）

### 5.1 单条目解析步骤（智能体按顺序生成代码）
- 选择具体子数据集名称并读取对应 `metadata`
- 逐条解码 `position`、`particle_type` 和 `context` 等字段
- 按窗口长度切片，生成训练样本或整轨迹评测样本
- 记录粒子数、窗口长度和边界范围等元信息
- 导出 `JSON + NPZ` 或图结构表示

### 5.2 多样本或多分量处理规则
- 不同子数据集、family、scene 或 split 必须分开处理，不允许混合统计量。
- 批处理前应先运行轻量字段审计，确认样本中不存在格式漂移或字段缺失。

## 六、校验规则与异常处理（判断条件+分支逻辑）

### 6.1 必做校验（代码必须实现，顺序执行）
- 位置维度必须与 `metadata` 定义一致。
- 粒子类型长度必须与粒子总数一致。
- 轨迹长度应满足窗口切片要求。

### 6.2 异常处理分支（固定捕获+处理逻辑）

| 异常类型 | 触发场景 | 代码处理逻辑 |
|---|---|---|
| `DataLossError` | 记录损坏 | 跳过坏记录 |
| `KeyError` | 缺少 `position` 或 `particle_type` | 标记坏样本 |
| `MemoryError` | 轨迹过大 | 使用滑窗流式导出 |

## 七、数据过滤与清洗规则（剔除条件清单）
代码必须在解析过程中实时过滤，禁止等全部数据读取完成后再统一做清洗。这样做可以减少内存占用，也能更快发现输入源中的结构性问题。

- 过滤空轨迹或粒子数过少样本。
- 按子集区分单材料、多材料或 3D 数据，避免混合。
- 过滤坐标越界或明显损坏的轨迹。
