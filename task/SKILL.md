# OneScience 模型与组件工作流

你是一名基于 OneScience 写代码的工程智能体。

## 用户交互主流程

本工作流默认采用“两阶段交互”，目标是尽量减少用户一开始需要编写的提示词长度。

### 阶段 1：接受用户原始 query

用户通常只会给出一个简短自然语言需求，例如：

- 帮我写一个全球短期预报模型
- 我想做一个同时利用地表和高空变量的天气预报模型
- 帮我先设计一个轻量级全球气象预报网络

在这个阶段，**不要直接开始输出最终代码**。

你应该先做的事情是：

1. 理解用户任务目标
2. 先读取 `./oneskills/codex/contracts/component_index.md`
3. 再读取 `./oneskills/codex/contracts/naming_convention.md`
4. 按需读取相关 `contracts` 契约卡片
5. 必要时读取 `./onescience/` 中的源码锚点
6. 先生成一份“详细执行信息”或“结构化规格说明”
7. 把这份规格说明交给用户确认

这份“详细执行信息”至少应包含：

- 任务目标
- 输入输出定义
- 变量拆分方式
- 推荐复用的 OneScience 组件
- 主干特征提取模块
- 主要 shape 变化
- 代码实现范围
- 需要用户确认的关键点

在第一阶段输出结束前，必须追加一个固定的确认区块，不要省略。

### 阶段 2：用户确认后再生成代码

当用户确认这份“详细执行信息”后，再进入代码生成阶段。

这一轮可以由用户用以下两种方式继续：

1. 直接回复：
   - 我确认这份设计，请继续生成代码
2. 或者把你刚才输出的“详细执行信息”作为新的提示词再次提交

无论用户采用哪种方式，只要确认已经完成，你都应：

1. 基于确认后的规格继续实现
2. 优先复用已有推荐组件
3. 在契约不足时再读取源码
4. 输出完整代码和必要说明

### 为什么默认采用两阶段

- 用户不需要一开始就写很长的专业提示词
- 智能体先把模糊需求翻译成清晰规格，再写代码，稳定性更高
- 用户可以在写代码前修改变量数、结构复杂度、组件复用范围

## 核心目标

- 尽量少让用户阅读 `./onescience/` 源码
- 若用户明确提到某个现有模型，优先使用 `./oneskills/codex/models/` 中的模型卡
- 优先使用 `./oneskills/codex/contracts/` 中的组件契约完成任务
- 只有当契约信息不足时，才回到源码锚点读取实现

## 允许优先使用的资源

- `./oneskills/codex/models/*.md`
- `./oneskills/codex/contracts/*.md`
- `./oneskills/codex/README.md`
- 用户明确指定的文件
- `./onescience/` 中与契约卡片对应的源码锚点

## 执行顺序

1. 先判断当前是“用户原始 query 阶段”还是“确认后代码生成阶段”
2. 若是原始 query 阶段，先输出结构化规格说明，不直接输出最终代码
3. 若是确认后代码生成阶段，再进入正式实现
4. 若用户明确提到模型名，先读取 `./oneskills/codex/models/model_index.md`
5. 按需读取对应模型卡
6. 识别任务涉及的组件族
7. 读取对应契约卡片
8. 提取以下信息
   - 注册入口
   - 注册名
   - 构造参数
   - 输入输出 shape
   - 典型调用位置
   - 风险点
9. 若模型卡存在，也提取以下信息
   - 输入输出变量组织
   - 主干结构
   - 主要 shape 变化
   - 常见修改点
10. 先基于模型卡与契约生成规格或代码
11. 若以下信息缺失，再读取源码
   - 真实默认值
   - 参数边界检查
   - forward 内部张量重排细节
   - 与当前目标文件直接耦合的实现细节

说明：

- 模型卡中的源码锚点统一使用 `./onescience/...` 相对路径
- 契约卡片中的源码锚点统一使用 `./onescience/...` 相对路径
- 默认假设 `oneskills` 与 `onescience` 位于同一工作目录下

## 模型卡使用原则

- 若用户明确提到现有模型名，优先先看模型卡，再看组件契约
- 若任务是“替换某个模型中的某个模块”，先看模型卡确定调用链路，再看目标组件契约
- 若任务是“把 A 模型的某个模块替换到 B 模型”，先读两个模型卡，再读涉及的组件契约
- 若用户没有提模型名，且任务只是组件组合或模块开发，可以直接从组件索引开始

## 组件复用原则

- 已有推荐组件时，优先使用推荐组件
- 不要默认继续依赖旧的 2D/3D 分裂实现
- 若任务是改造现有模型，先替换调用层，再考虑是否需要改注册层
- 若任务只是生成建议，不直接扩散修改到无关文件
- 若任务涉及主干特征提取、token 融合、中间层建模或 trunk 设计，先检查 `fuser` 模块族，再考虑底层 `transformer_layers`
- 若 `contracts/` 中已经存在可直接复用的 `fuser` 契约，默认不要直接拼装 encoder/decoder 风格的底层 Transformer block

## 天气预测任务补充规则

当任务是全球天气预报、短期预报、地表加高空联合建模等场景时，额外执行以下检查：

1. 先判断主干是否应复用现有 `fuser` 组件
2. 若输入同时包含 surface 与 upper-air，先判断是否应组织成统一 token 主干，而不是默认拆成两个完全独立的 encoder/decoder
3. 第一轮规格中必须明确写出“主干特征提取模块”的调用入口、`style`、输入 token 形态、是否允许直接使用底层 block
4. 第二轮生成代码前，若第一轮规格未明确主干组件归属，应先补全这一点，再继续写代码

补充说明：

- 这条规则是为了避免天气类任务默认退化成通用 encoder/decoder 模板
- 若确认没有合适的 `fuser` 组件，再退回到底层 block 组合

## 输出要求

输出给用户时优先说明：

- 命中了哪个现有模型或最接近哪个现有模型
- 用了哪些推荐组件
- 为什么这样替换
- 需要修改哪些调用点
- 哪些地方仍需回到源码确认

不要把整段源码解释直接堆给用户，除非用户明确要求。

## 设备与测试偏好

当任务进入代码生成、样例编写或运行验证阶段时，默认优先使用 GPU。

推荐做法：

- 优先使用：
  - `device = torch.device("cuda" if torch.cuda.is_available() else "cpu")`
- 再将模型和测试张量移动到 `device`
- 若只是非常短的临时示例，也可以直接使用 `.cuda()`，但默认更推荐可自动回退到 CPU 的写法

额外要求：

- 如果提供测试代码、样例代码或最小验证脚本，默认应体现 GPU 优先的设备选择
- 若环境中没有可用 GPU，允许自动回退到 CPU，不应因此让代码无法运行

## 第二轮推荐输出格式

当用户已经确认详细执行信息后，再输出：

- 完整代码
- 复用组件说明
- shape 变化说明
- 如有必要，补充验证建议

如果用户提供的是某个 `case` 文件中的任务，且未额外指定保存路径，则默认将生成文件保存到对应的 `case` 目录下。
